# (collab join)

Register one joinable participant in the active collaboration record from command-owned role JSON.

## Trigger

**Dispatch:** `(collab join --role <role>)` — routing-only command form; not a shell command.
**Search phrases:** collab join, join collaboration record, register collab role

## Steps

1. Read [invariants.md](../../../commands/collab/reference/invariants.md) before executing; call the relevant helper fresh and do not trust prior reads from conversation context (Invariant #4). Resolve the target collab with **Registry targeting** in **Notes**.
2. Resolve `<role>` from `--role <role>`. If missing, **ABORT**: `--role <role>` is required.
3. Read `../../../commands/collab/reference/roles/<role>.json`. If unreadable, **ABORT**: role file unreadable; name the expected path.
4. Validate the role JSON against **Role schema** in **Notes**. If invalid, **ABORT**: invalid role JSON; name the failed field.
5. Read the resolved registry and the resolved transcript path. If either is unreadable, **ABORT**: record unreadable; name the path.
6. If the registry status is `closed` or `archived`, **ABORT**: record is closed.
7. Append `<role>` to the registry `participants` list when missing. If the role already exists, leave the registry roster unchanged.
8. If the registry `turnOrder` is empty, seed it from `participants`, excluding `reviewerRole` when `reviewerMode` is `last-in-convergent-phases`. If the role is new, `turnOrder` exists, and the role is not that reviewer, append the role at the end.
9. Declare the active runtime harness identity as `--agent-id` before invoking the helper, using the precedence in [agent-id.md](../../../commands/collab/reference/agent-id.md). The value must come from the current harness or execution environment. Do not copy from role files, prior collab records, test fixtures, examples, or documentation. If the harness does not expose a usable identity, pass the literal `unknown`. This step applies to all roles, including the moderator.
10. Call `commands/collab/engine/registry.py join-participants <target> <role> --agent-id <agentId>`. The helper validates all role files, enforces a non-blank `agentId`, computes the next registry participants and turn order in memory, renders the full participants table from that state, then writes registry and transcript together. Participation entry point: `commands/collab/engine/registry.py speak-state --resume <target> <role>`. The first output line is `NEXT: Run (collab show policy) before first speak.`; display it before the participant roster output. Abort if any role file is missing or unreadable, or if `agentId` is missing or blank, before either file is written.
11. Sync the Turn order cell in the transcript state table from registry `turnOrder` after any registry or transcript participant update.
12. Stop after updating registry and transcript participants. Do not write a phase contribution.

## Notes

- **Parameters:** target collab slug, id, or numeric `#N` as the first positional token after `join`; when absent, resolved per **Registry targeting** in **Notes**. `--role <role>` — required role key.
- **Registry targeting:** Resolve the target collab from the resolved registry, using `commands/collab/engine/registry.py` as the shared helper. When the first token after the route is present, treat it as a collab slug, id, or stable numeric position. Otherwise use `activeCollabId`. If the registry is unreadable or invalid, the token does not match any entry, or `activeCollabId` is empty, **ABORT**: registry target unavailable; name the registry field or token.
- **Role schema:** Defined in [platform/standards/agent-role.md](../../../platform/standards/agent-role.md). Required fields: `key` (must match `<role>`), `displayName`, `concerns` (non-empty array). Optional field: `prohibitions` (string array; principle-level behavioral constraints, not a runtime enforcement list). For per-role join-time model recommendations, see [`agent-model.md`](../../../commands/collab/reference/agent-model.md).
- **Participant row format:** `| <#> | <key> | <displayName> | <agentId> | <concern>; <concern> |`. The table header is `| # | Key | Role | Agent | Responsibilities |`. The `agentId` value is the joining agent's active runtime harness identity declared in Step 9 on first join; see **`agentId`** in **Notes**.
- **`agentId`:** The joining agent's active runtime harness identity, recorded once at join time. Declare it with the shared vocabulary and precedence in [agent-id.md](../../../commands/collab/reference/agent-id.md). The token `unknown` is reserved for harness-inaccessible identity; free-form alternatives (`UNKNOWN`, `unspecified`, `n/a`) are rejected by the helper. Do not copy `agentId` from role files, prior collab records, test fixtures, examples, or documentation.
- **Trust-model scope:** `agentId` enforces presence and freshness, not truthfulness. It is an honest-effort marker, comparable to a git commit author. It is not an authentication signal; a determined agent can write any string.
- **Capture-at-join:** Identity is recorded once per role per collab at join time. No speak-time revalidation. If the same role re-runs join with a different supplied `agentId`, the helper keeps the recorded at-join value and emits an `IDENTITY-WARN:` advisory instead of overwriting registry state.
- **Model drift:** When a session spans a model upgrade, the registry retains the at-join value. This is accepted behavior, not a defect.
- **Helper enforcement boundary:** The helper enforces presence, the exact `unknown` token, and whitespace stripping. The precedence preference (model ID over harness name) is an agent obligation; the helper cannot verify which tier an agent chose.
- **Pending reviewer display:** When `reviewerRole` is set in the registry but that role is not yet in `participants`, the transcript **Reviewer** section marks the role as `(pending)`. The `(pending)` state is registry-owned — derived from `reviewerRole` not being in `participants` — not from transcript prose. Once the reviewer joins via `(collab join)`, the `(pending)` label is replaced with the active reviewer notice.
- **Pending-reviewer speak gate:** A pending `reviewerRole` blocks all participant speaks. `speak-state` aborts before turn-order checks when `reviewerRole` is set and absent from `participants`. The reviewer must join via `(collab join --role <reviewer>)` or be cleared via `(collab set reviewer --clear)` before any participant may contribute.
- **Role data boundary:** Joinable role JSON files live under `commands/collab/reference/roles/` as shared command-owned data, not route playbooks. Projector metadata under `commands/collab/reference/projectors/` is projection-only and must not satisfy `(collab join --role <role>)`.
- **Section-targeting convention:** A `render-*` helper owns exactly one transcript section identified by its visible heading, takes registry state and role JSON as its only inputs, and fails before any write when input validation fails.
- **`render-participants` boundary:** `render-participants` is a repair and diagnostics subcommand. Do not call it in route prose; the normal join path uses `join-participants`.
- **Recovery path:** If `join-participants` aborts due to a missing or unreadable role file, remove the orphaned key from registry `participants` or restore the role file, then re-run `(collab join)`.
- **One-time normalization:** On the first `(collab join)` after `join-participants` is deployed, the full participants table is regenerated from registry state; any hand-edits that diverged from the registry are replaced. This is expected behavior.
- **After joining:** The helper emits `NEXT: Run (collab show policy) before first speak.` as its first output line. Run `(collab show policy)` before your first `(collab speak)` to read the gate rules and reviewer contract for this collab. The machine-verifiable resume signal is: `commands/collab/engine/registry.py speak-state --resume <target> <role>`. Run this after any `/compact`, agent swap, or subagent return to re-establish collab context before writing.
- **Sync contract compliance:** Step 11's Turn order cell sync is prose-rendered; `join-participants` renders the participants table but does not own the state-table Turn order cell. This is declared under the sync contract in [`platform/standards/route-invariant.md`](../../../platform/standards/route-invariant.md).
- **Advisory output contract (join):** After a successful join, the helper emits advisory lines in this fixed order: `NEXT:` first (imperative routing, e.g. `Run (collab show policy) before first speak.`), `EFFORT:` second (carrying the current `activePhase`, joining role, effort level, and scale phrase derived from `commands/collab/reference/agent-effort.json`), `IDENTITY:` third (the at-join `agentId`, so the moderator can verify the recorded identity without reading the registry), `RESUME:` fourth (`commands/collab/engine/registry.py speak-state --resume <target> <role>`, so the joining agent can re-establish context after any `/compact` or agent swap), `TRANSCRIPT:` fifth (`commands/collab/engine/registry.py transcript-view <target> <phase> --raw`, so the joining agent can read active-phase lifecycle contributions without separate discovery). `IDENTITY:` reflects the captured at-join identity only; it does not overwrite registry on cross-harness re-join. If a re-join supplies a different `agentId`, the helper emits `IDENTITY-WARN:` after `IDENTITY:` and keeps the registry value unchanged. No `EFFICIENCY:` line is emitted at join time; the join action does not cross a lifecycle boundary. On gate failures, an additional `RESUME: commands/collab/engine/registry.py speak-state --resume <target> <role>` is emitted as a terminal advisory line so agents can re-establish state without consulting documentation. Consumers must parse by prefix label, not by line index; fixed ordering is a readability convention only.
