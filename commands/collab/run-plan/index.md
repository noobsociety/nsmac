# (collab run plan)

Implement the action-plan items assigned to the executing agent's role in a collaboration record.

## Trigger

**Dispatch:** `(collab run plan)` — routing-only command form; not a shell command.
**Search phrases:** run collab action plan, complete collaboration tasks, collaboration execution

## Steps

1. Read [invariants.md](../../../commands/collab/reference/invariants.md) before executing; call the relevant helper fresh and do not trust prior reads from conversation context (Invariant #4). Resolve the target collab with **Registry targeting** in **Notes**.
2. Read the resolved registry and the resolved transcript. If either is unreadable, **ABORT** (agent-honor-system): record unreadable; name the path. The `execution` and `execute-spawn` helpers tolerate a missing transcript rather than aborting, so this read-time guard is agent-enforced.
3. If the registry status is `closed` or `archived`, **ABORT**: closed collaboration records cannot be executed. The `execution` helper backstops this with `record is closed`.
4. Resolve the active phase from registry `activePhase`. If missing or unknown, **ABORT**: active phase missing in metadata. The `execution` helper validates the phase enum on load and rejects an unknown value with `collab activePhase must be one of [...]`.
5. If the active phase is not `Completion`, **ABORT**: `(collab run plan)` is valid only when registry `activePhase` is `Completion`. The `execute-spawn` helper enforces the same gate before any subagent scope check, aborting with `execute-spawn is valid only in Completion`.
6. Resolve the executing role from the registry participants list. Match the current agent to a registered participant. If no match, **ABORT**: role not registered; run `(collab join --role <role>)` first. The `execution` helper backstops this with `execution role must already be a participant: <role>` when the executing role is not yet a participant.
7. Read `## Action Plan`. Collect all unchecked checklist items whose label begins with `**<role>:**` matching the resolved role. If none found, report that no assigned items remain and stop.
8. If Step 7 found no unchecked assigned items and an execution-history item matching `**<role>:** completed` already exists, report that the role's execution is already complete and stop. If Step 7 found unchecked assigned items, continue execution; a later successful `execution` helper write replaces the role's prior execution metadata.
<!-- abort: run-plan-requires-chain -->
9. For each collected item, check `## Handoff` for a corresponding entry containing `_requires: #N_` or `_requires: #N #M_`. Parse every required item number listed in that field. If a required item number is found, verify its checkbox in `## Action Plan` is checked (`[x]`). If any required item is unchecked, **ABORT** before any file write with a structured message: executing role, blocked item number, required item number(s), and required item state (`unchecked`). This is the "perform or halt" contract: execution either runs or aborts with a concrete reason — never partial. The `execution` helper is the enforcement backstop: it refuses to record `completed` while unchecked assigned Action Plan items remain, with `execution completed blocked for role <role>: <N> unchecked assigned Action Plan item(s) remain; loop target: Handoff for missing execution evidence`.
10. Ensure `## Completion` contains `**Execution history**`.
11. If using subagents for implementation work, read the executing role's structured Handoff state with `commands/collab/engine/registry.py handoff-state <target> <role>` and assign subagent scopes from its `writeScope` values. For each subagent, call `commands/collab/engine/registry.py execute-spawn <target> <role> --scope <path> --sibling-scope <path>...` to validate that worker's assigned scope against the other declared scopes and the role's declared Handoff `writeScope`. After that subagent returns, call the same helper with `--returned-path <path>` for every returned path and reject any patch outside that worker's assigned scope. **Subagents implement scope within the executing role's own assigned items only. Action-plan items labeled for another role (`**<other-role>:**`) are never delegated to a subagent — they are that role's responsibility and must not be touched.**
12. Implement each collected action-plan item in source, in the order listed.
13. Run scoped validation for the executing role. Minimum scoped validation is `./platform/tooling/audit.sh`; add targeted tests for touched behavior when available. Do not run `./tests/run.sh` for ordinary role-scoped completion. If the collected item is the terminal full-suite Action Plan item, run the full sequence instead: `./platform/tooling/audit.sh` + `./tests/run.sh`.
14. Check every completed role-scoped checklist item in `## Action Plan` as `[x]` before reporting success.
15. Mirror execution state in the registry `execution` object for the resolved collab by calling `commands/collab/engine/registry.py execution <target> <role> completed <timestamp> --agent-id <agentId> --validation-result passed --validation-scope scoped --touched-path <path>...` (or `failed` with `--validation-result failed`), including all touched paths. Use the active execution harness identity for `--agent-id`; this lets participant verification annotate remediation when it is performed by a different agent. The helper rejects touched paths outside the role's structured Handoff `writeScope`. Use `--validation-scope full` only for the terminal full-suite item and `--validation-scope deferred` only when a role records no local validation because its checks are intentionally owned by a later terminal item. Pass `--auto-close` only after the terminal full-suite owner has completed and all assigned roles are complete. The helper emits a `NEXT:` advisory after writing; see **NEXT advisory contract** in **Notes**.
16. After recording execution, call `commands/collab/engine/registry.py participant-verify-state <target> <role>` to check whether participant verification is immediately due for the executing role. If the output shows `readyToVerify: true` and `nextRole` equals the executing role, run `(collab participant verify)` for the executing role before advancing. Participant verification is a Completion-phase obligation for roles assigned to it; when it is due, it belongs in this route's execution sequence and must not be deferred to a separate invocation.
17. Evaluate the **Auto-close on completion** rule in **Notes**. If every non-moderator assigned role, including the terminal full-suite owner, has a completed execution entry, update the registry status to `closed`, clear `activeCollabId` when it points at this collab, update the transcript Status cell to `closed`, and generate the default close summary under `## Completion`.
18. Append the next numbered item as `<n>. **<role>:** completed YYYY-MM-DD HH:MM — validation passed; <scope>; <N> paths.` after validation passes, where `<scope>` is `scoped`, `full`, or `deferred` and `<N>` is the count of touched paths. If validation fails, append `<n>. **<role>:** failed YYYY-MM-DD HH:MM — validation failed: <failed command>; <scope>; <N> paths.` before reporting.
19. Report all changed files and validation results. Stop.

## Notes

- **Parameters:** target collab slug, id, or numeric `#N` as the first token after `run plan`; when absent, resolved per **Registry targeting** in **Notes**.
- **Registry targeting:** Resolve the target collab from the first token after the route, falling back to `activeCollabId` when absent. The resolution algorithm and abort contract are owned by **Target resolution** in [`platform/standards/route-invariants.md`](../../../platform/standards/route-invariants.md); this route does not restate them.
- **Completion-only guard:** `(collab run plan)` must refuse `Audit`, `Discussion`, `Conclusion`, `Action Plan`, and `Handoff`.
- **Item scoping:** Execution selects only checklist items whose label begins with `**<role>:**` matching the executing role. Items labeled for other roles are skipped without modification.
- **Executable Action Plan shape:** Executable items are flat checklist lines beginning with `**<role>:**`. Nested checklist items, role-group headings, or unlabeled child tasks are not executable assignments. Action-plan items must not assign running the full test suite (`./tests/run.sh`) as a per-agent completion step; the full-suite run belongs to the terminal execution turn only.
- **Dependency parsing:** A handoff `requires` field may list one or more action-plan item numbers separated by spaces, such as `_requires: #1 #2 #3, next: ..._`. Treat every listed number as required. See [`handoff-shape.md`](../../../commands/collab/reference/handoff-shape.md) for the Handoff field set (`writeScope`, `validationCommands`) that also governs `execute-spawn` scope consumption.
- **Current behavior source:** The previous per-agent full-suite expectation came from this route's explicit validation sequence, not from `commands/collab/engine/registry.py`; the helper records execution metadata but does not run validation.
- **Terminal suite owner:** The full test suite (`./tests/run.sh`) is owned by one terminal suite owner in `Completion.execution`, not by each role independently. When `reviewerRole` is configured, the reviewer role may be assigned this terminal full-suite Action Plan item; that assignment is execution work and is separate from the reviewer's later `Completion.verification` seal/verdict obligation. When no reviewer exists, the fallback owner is the final non-moderator assigned role in `turnOrder`; that owner must receive an explicit terminal full-suite Action Plan item before close can be treated as complete. Per-role `(collab run plan)` execution must not invoke `./tests/run.sh` as a per-agent completion check; scoped smoke validation suffices for individual role turns. See `commands/collab/init/index.md` → **Terminal suite owner authoring** for the authoring-time requirement.
- **Late-failure recovery:** If the terminal full suite fails and ownership is clear, rewrite or add the failing role's Action Plan item and re-run that role before repeating terminal validation. Re-enter Audit only when the failure invalidates the agreed architecture or exposes cross-role scope uncertainty.
- **Malformed items:** Treat unchecked checklist items without a `**<role>:**` label as out of scope unless a future route spec defines another assignment form.
- **Action Plan label contract:** Checklist items with a `**<role>:**` label are owned assignments; unlabeled `- [ ]` items are unowned and skipped by all role executions. Use labeled items for executable work only.
- **Layer-1 enforcement:** `commands/collab/engine/registry.py` rejects recording `execution.<role>.status = completed` when unchecked `**<role>:**` Action Plan items remain. The abort message names the role and unchecked item count: `"execution completed blocked for role <role>: <N> unchecked assigned Action Plan item(s) remain; loop target: Handoff for missing execution evidence"`. Recovery: mark all assigned items `[x]` in the transcript, then re-run `(collab run plan)`.
- **Touched-path enforcement:** When structured Handoff state exists for the executing role, `commands/collab/engine/registry.py execution` rejects any `--touched-path` outside `handoff.roles.<role>.writeScope` with `"execution touched path outside declared writeScope: <path>"`. Recovery: reopen Handoff and revise scope, or remove/revert the out-of-scope change before recording execution.
- **Auto-close on completion:** `(collab run plan)` owns the completion close trigger because it owns `execution.<role>`. Close only when every non-moderator assigned role has a completed execution entry; failed or missing entries leave the record open.
- **`record_execution` wholesale-replace contract:** A call to the `execution` helper for a role replaces that role's full execution state. Any `touchedPaths` not re-declared in the new call are silently dropped from the coverage aggregate the seal reads. There is no partial-update or append path; every `(collab run plan)` invocation for a role must re-declare the complete set of that role's touched paths for the current round. After a reopen, the coverage carry-forward for prior rounds is supplied by the reopen snapshot — not by re-submitting prior paths through `record_execution`.
- **Re-execute:** To rewrite the last execution record in-place, use `(collab rewrite execution)`. See [rewrite execution](../rewrite-execution/index.md).
- **Parent-owned subagent integration:** Subagents may implement bounded work only under the executing parent role — and only that role's own assigned action-plan items; items labeled for another role (`**<other-role>:**`) are that role's responsibility and are never delegated to a subagent. The parent declares one assigned write scope per subagent, uses `commands/collab/engine/registry.py execute-spawn <target> <role> --scope <path> --sibling-scope <path>... [--returned-path <path>...]` as the per-subagent write-scope gate, rejects returned paths outside that subagent's assigned scope, computes touched paths from the accepted parent patch after integrating returned work, runs repository validation itself, and records execution status from the parent role only. `execute-spawn` itself enforces that the active phase is `Completion`, so it must be invoked only from within this route's flow. A subagent never authors a collab turn and never mutates registry or transcript state independently.
- **Execution boundary:** The route implements only action-plan items assigned to the current role. Its only lifecycle side effect beyond implementation is the auto-close trigger when all assigned execution is complete. Use `commands/collab/engine/registry.py write-guard <route> <path>...` as the shared helper boundary: non-execution collab routes may write only under `registry.json`, `registry.json.lock`, or `records/` inside the user-scope collab state root, while the `execution` helper records role, status, validation result, and touched paths in execution history and registry execution metadata.
- **NEXT advisory contract:** After a successful registry `execution` helper call, including calls made by `(collab run plan)` and `(collab rewrite execution)`, the helper emits a `NEXT:` advisory naming the first pending workflow action from post-write registry state. `NEXT:` is a forward pointer: it must name the next required action to advance the workflow, not the action just recorded. When all assigned execution roles are complete, the advisory advances to reviewer verification or seal as appropriate; it must not name a participant-verify, seal, or assessment step while any execution role remains pending. Execution advisory routing must go through `next_line_after_execution` as the single source; a separate advisory path in either route is a defect.
- **Post-state resume signal:** Step 16 calls `participant-verify-state` to gate the participant-verification check. After the full sequence completes (including participant verification when due), the collab is either still open (peer execution pending) or auto-closed. Run `commands/collab/engine/registry.py speak-state --resume <target> <role>` to confirm the current state before any further action.
- **Sync contract compliance:** Steps 10, 14, and 18 write transcript prose (section initialization, completed/failed markers, and Action Plan checkbox updates) without a helper. The exemptions are declared under the sync contract in [`platform/standards/route-invariants.md`](../../../platform/standards/route-invariants.md).
