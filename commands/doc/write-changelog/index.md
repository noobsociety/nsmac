# /doc write changelog

Create or update `CHANGELOG.md`, `NEWS.md`, `HISTORY.md`, `RELEASE_NOTES.md`, and similar changelog files with a curated six-category structure and git-grounded entries.

## Trigger

**Slash:** `/doc write changelog`
**Signature:** `/doc write changelog <atomic | squash>`
**Prose dispatch:** `(doc write changelog <atomic | squash>)` — prose routing hint; not a terminal command.
**Search phrases:** `changelog atomic`, `update changelog atomic`, `changelog squash`, `update changelog squash`

## Steps

1. Resolve `<atomic | squash>` from the first argument. **ABORT**: `<atomic | squash>` is required.
2. Resolve **path** (user-supplied, else default repo-root changelog, else **ask once**).
3. Open the target or create from **Document skeleton** in **Notes** (H1 = **Title by filename**).
4. Run `git rev-parse --show-toplevel`; on failure or no git, note once and skip git sections.
5. Resolve **Replay source** in **Notes** before writing entries.
6. **Operate** — run replay for the resolved mode per **Notes**; then either log under `[Unreleased]` or cut a release (`## [<version>] - YYYY-MM-DD`, ask once for version, confirm date, clear `[Unreleased]`, drop empty `###`); open new `[Unreleased]` after cutting.
7. **Validate:** no duplicate facts across categories or releases; no empty `###`; newest dated `##` below `[Unreleased]`; voice matches **Entry rules**.

## Notes

- **Route (atomic vs squash).** `atomic` → one bullet per non-merge commit in the resolved source range, chronological within each `###`. `squash` → one chosen squash commit; one bullet per accepted body line per the **Squash procedure**.
- **Parameters:** `<atomic | squash>` — git replay mode (required). `atomic`: one bullet per commit in the resolved source range. `squash`: one bullet per accepted body line of a chosen squash commit. For "since last release" or a tag, anchor `git log` from that tag when named.
- **Dependencies:** [markdown-workflow](../../../platform/standards/markdown-workflow.md), [markdown-workflow](../../../platform/standards/markdown-workflow.md); `/doc write changelog` directives take precedence over those rules on changelog paths. If either is unreadable when needed, **ABORT** per **`context-gate.md`**.
- **Post-run report:** After each run, emit a structured summary in this exact form — omit the "Squash OID" line for atomic mode: `Updated <path>. Mode: <atomic|squash>. Git range: <YYYY-MM-DD> → <YYYY-MM-DD>. [Squash OID: <sha>.] Ordering: <rule applied, e.g. "chronological within each category" or "body-line order of selected commit">. Sections written: ### <Category> (<N> items)[, …].` No unsolicited coaching unless asked.
- **Voice:** Shared markdown defaults; changelog entry tense overridden here. **`[Unreleased]`:** imperative or simple present. **Dated / `[YANKED]`:** past tense.
- **Document skeleton:** H1; short intro explaining the file purpose plus optional SemVer line; `## [Unreleased]`; then dated `##` newest first. **Title by filename:** `CHANGELOG*` → `# Changelog`; `NEWS.md` → `# News`; `HISTORY.md` → `# History`; `CHANGES*` → `# Changes`; `RELEASE_NOTES*` → `# Release notes` or match project. Dates use `YYYY-MM-DD`. Only non-empty `###`. List lines respect ~120 chars when caps exist elsewhere.
- **Replay source:** Resolve a concrete git range before writing. If the target file does not exist, default to full repository history from the first relevant commit through `HEAD` unless the user supplies a narrower anchor. If the first relevant commit cannot be determined, ask once before writing. Existing changelogs default to the week window below unless the user names another anchor or the file's coverage preamble requires reconciliation. Partial ranges must use **Coverage preamble**.
- **Coverage preamble:** Add a preamble when the changelog does not cover full repository history. Trigger: (a) the target file did not exist and the user supplied a range narrower than the repository root, or (b) the existing file's stated coverage does not reach the repository's first relevant commit. Placement: a blockquote immediately before `## [Unreleased]`, after the document intro. Template — substitute `YYYY-MM-DD` values, use verbatim otherwise: `> **Coverage:** Commits from YYYY-MM-DD to YYYY-MM-DD. History before YYYY-MM-DD is not included.` Where the first date is the oldest included commit date and the second is the newest. Do not add the preamble when full repository history is covered.
- **Replay contract:** Use the resolved source range for both modes. Sort `atomic` by committer date oldest to newest within each `###`. Preserve accepted squash body-line order for `squash`. Map categories with **Commit authority**. Dedupe by normalized category plus bullet text plus source OID. Apply **Bullet form** after mapping. Keep drift bullets with no matching git line and note drift. Report range, mode, ordering, destination sections, and squash OID when applicable with **Post-run report**.
- **Categories (omit empty):** **Added** · **Changed** · **Sunset** · **Removed** · **Fixed** · **Security**. Map: new→Added; behavior/perf/tuning→Changed; regressions→Fixed; deletes→Removed; vulnerability-related fixes→Security; sunset notices→Sunset. Optional **Performance** is non-standard; prefer **Changed** unless the file already splits it. Curate entries; do not paste raw logs.
- **Entry rules:** One line per change; say what changed and why; **`[Unreleased]`** imperative or present; dated past tense; yanked bullets start “Yanked because …”; sentence case; no emoji; no vague “misc”; no duplicate across categories or releases; no leading category verb in bullet text.
- **Bullet form:** Category headings (`### Added`, `### Fixed`, etc.) supply the category label; bullet text supplies the change description only. Never prefix bullet text with the category name or its verb form — under `### Added`, write “Support for X” or “Add support for X”, not “Added support for X” or “Added: support for X”. Voice follows the section: `[Unreleased]` bullets use imperative or simple present; dated-release bullets use past tense. This rule applies identically to both atomic and squash modes with no per-run variation.
- **Commit authority:** No bullet without matching commit unless user states it; use `git log` / `git show`. Conventional `type(scope): description` (lowercase; imperative in git). **Type → `###`:** `feat`→Added; `fix`→Fixed; `perf`→Changed; `docs`/`style`/`refactor`/`chore`/`ci`/`test`→Changed unless **removal heuristic** (description contains remove/drop/delete, case-insensitive, or clearly retires capability)→**Removed**. Security-related `fix`→**Security** when vulnerability-related or user asks. **Replay text:** use the description after `type(scope):`; strip `fixup!`/`squash!`; keep wording and order; apply entry voice and sentence case (capitalize first letter) without paraphrasing.
- **Week window (local TZ):** Monday–Sunday; git `--since='<this Monday> 00:00:00' --until='<next Monday> 00:00:00'` (exclusive second bound). **Dated `##` for week:** newest committer date `%cs` in window. **Empty window:** note once; sync `[Unreleased]` only; do not invent week `##`. **Reverse drift:** bullet with no matching git line — keep bullet and note drift.
- **Squash vs atomic:** Same resolved source range, same type→`###` map, same removal heuristic. **Squash:** one chosen commit; parse subject then body lines (`^(?:-\s+)?[a-z]+(?:\([^)]+\))?:\s+.+$`) as candidates; run `git show --name-status <oid>` for the selected commit; accept only distinct candidates whose conventional type/scope can be reconciled with touched paths or repo-level metadata; flag or reject candidates whose type/scope does not match touched paths. Preserve accepted body-line order, then enforce entry voice and sentence case. **Atomic:** each non-merge in range with conventional subject; subject only (no body lines); one bullet per commit; chronological order within `###`. **Outside range** → `[Unreleased]` with correct `###` and voice. **Squash procedure:** `git rev-parse`; user `<ref>` or walk in-range newest-first, `git show` each candidate once, first with ≥1 matching line in subject or body; retain `%H` `%cs` `%B`; run `git show --name-status <oid>` once for the selected OID; title dated `##` from newest `%cs` in range; dedupe subject repeated in body; reconcile; never `git show` same OID twice. **Atomic procedure:** `git log <range> --no-merges --reverse --pretty=format:'%h%x09%cs%x09%s'`; one dated `##` (newest `%cs`); omit unparsable subjects with note; reconcile reverse drift. **Compact log:** `git log <range> --no-merges --pretty=format:'%h %s'`.
- **Release ops:** Cut → new `##`, past tense, clear `[Unreleased]`; hotfix thin `##`; yanked `## [ver] - date [YANKED]` with “Yanked because …”.
- **Unreleased vs dated section:** When the working tree is **clean** at the end of a squash run and `[Unreleased]` is non-empty, automatically cut a dated release — move all `[Unreleased]` bullets to a new `## [YYYY-MM-DD] - YYYY-MM-DD` heading (past tense, newest committer date in the replay range), then open an empty `[Unreleased]` above it. Do **not** ask before cutting; a clean tree is the cut signal. If the working tree is **dirty**, log under `[Unreleased]` (imperative or present) and do not cut. Explicit `cut` or tag requests override both conditions per **Release ops**.
- **List order in `[Unreleased]` and dated sections:** `atomic` is **chronological** within each `###` (oldest to newest in the week window, unless a task specifies **reverse**). `squash` **replay** lines follow the **body line order** of the chosen commit for those lines only. Merged, reconciled, or long-lived buffers may be **sorted alphabetically** per `###` for readability (declare which rule you used; do not interleave `feat` and `chore` without a mapping pass).
- **Anti-patterns:** Past tense under `[Unreleased]`; category-verb bullet prefixes; inferring squash/atomic without user; trusting changelog over `git`; `git show` same OID twice in squash; paraphrasing commits; merging body lines; vague bullets; empty `###`; log dumps; dropping drift bullets silently; mixing modes in one run.
